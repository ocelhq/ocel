# PRD: Project management in the dashboard

Status: ready-for-agent

## Problem Statement

Organizations using Ocel can currently only group things at the Organization level — there's no way to represent "these resources and this application belong together as one deployable unit" within an org. As soon as an org has more than one thing being built or deployed, there's no boundary narrower than the whole organization to organize them under.

## Solution

Introduce a **Project**: a named, uniquely-slugged unit that belongs to exactly one Organization and will (in a future iteration) group the Resources and Applications deployed together. This PRD covers the Project entity itself and a secured CRUD API for managing it from the dashboard — laying the foundation other resources will attach to later.

## User Stories

1. As an organization member, I want to create a new Project within my organization, so that I can group related resources and applications logically before deploying them.
2. As an organization member, I want to give my Project a human-readable name, so that I can recognize it easily in the dashboard.
3. As an organization member, I want to give my Project a URL-safe slug, so that it can be referenced predictably in URLs and API calls.
4. As an organization member, I want the system to reject a Project slug that's already used by another Project in my organization, so that I don't accidentally create ambiguous references.
5. As an organization member, I want to use a slug that's already taken in a *different* organization, so that my org's naming isn't limited by unrelated orgs' choices.
6. As an organization member, I want to add an optional description to my Project, so that teammates understand its purpose without guessing from the name/slug alone.
7. As an organization member, I want to list all Projects belonging to my organization, so that I can see everything my team has set up.
8. As an organization member, I want to view a single Project's details by its id, so that I can inspect or link to it directly.
9. As an organization member, I want to rename a Project after creation, so that I can fix a typo or reflect a change in purpose.
10. As an organization member, I want to change a Project's slug after creation, so that I can correct it if it no longer matches the project's name.
11. As an organization member, I want to update a Project's description, so that I can keep documentation current.
12. As an organization member, I want to delete a Project I no longer need, so that my organization's Project list stays clean and relevant.
13. As an organization member, I want all Project API requests to require me to be signed in, so that unauthenticated users can't view or modify any organization's projects.
14. As an organization member, I want Project list/create requests to operate on whichever organization is currently active in my session, so that I don't have to specify the organization explicitly every time.
15. As an organization member, I want to view/update/delete any Project belonging to any organization I'm a member of, regardless of which org is currently "active" in my session, so that switching my active org doesn't lock me out of projects in my other orgs.
16. As an organization member, I want the system to reject my request to access a Project belonging to an organization I'm not a member of, so that other organizations' Projects stay private.
17. As an organization member, I want to know who created a Project, so that I can ask the right person questions about it.
18. As an organization owner, I want deleting my organization to automatically remove all of its Projects, so that no orphaned Project records are left behind.
19. As an organization member, I want a clear validation error when I submit an invalid Project name or slug (e.g. empty name, badly formatted slug), so that I understand what to fix before resubmitting.
20. As an organization member, I want a clear conflict error (not a generic failure) when I try to create/rename a Project into a slug that's already taken in my org, so that I understand exactly what went wrong.
21. As a future Ocel CLI user, I want Projects to exist as a stable API-addressable resource, so that a future `ocel init`/`ocel dev` can associate a local codebase with a specific Project id (implementation out of scope here, but the API shape shouldn't preclude it).

## Implementation Decisions

- **New `project` table** (Drizzle, kept in a schema file separate from the generated `auth-schema.ts`, since that file is regenerated by `pnpm --filter web auth:generate` and must not be hand-edited):
  - `id`: UUIDv7, generated in application code via the `uuidv7` package (not Postgres's default v4 `gen_random_uuid()`, and not Better Auth's own ID generator, since Project isn't a Better-Auth-managed table).
  - `organizationId`: FK -> `organization.id`, `onDelete: cascade` (matches the existing `member`/`invitation` FK pattern).
  - `name`: text, 1-100 characters, no charset restriction.
  - `slug`: text, regex `^[a-z0-9]+(-[a-z0-9]+)*$`, 1-63 characters. **Unique per `(organizationId, slug)`** — not globally unique like `organization.slug`. Editable after creation.
  - `description`: text, nullable, optional.
  - `createdBy`: FK -> `user.id`, nullable, `onDelete: set null` (project survives if the creating user's account is later deleted).
  - `createdAt` / `updatedAt`: standard timestamp columns.
  - Hard delete only — no soft-delete/archive column.
- **Schema wiring**: add a schema barrel re-exporting the auth schema + the project schema; update the Drizzle config's `schema` path to a glob and the db client's schema import to the barrel (both currently hardcode the single generated auth-schema path).
- **New session/membership helper** (first of its kind in the repo): resolves the current Better Auth session from request headers, reads the caller's active organization id, and — critically — independently verifies membership rows rather than trusting `session.activeOrganizationId` at face value (that column has no FK constraint in the schema).
- **New validation module** (zod schemas for create/update payloads, enforcing the slug regex and name length bounds); introduces `zod` as a new `apps/web` dependency (none exists today).
- **API contract** (Next.js route handlers under the projects API path):
  - `POST /api/projects` — create; scoped to the caller's active-session organization; any org member may call; 201 on success, 400 on validation failure, 409 on slug conflict, 401 if unauthenticated or no active organization.
  - `GET /api/projects` — list all Projects in the caller's active-session organization; unpaginated; 200.
  - `GET /api/projects/{id}` — fetch one Project; authorized against the **project's actual** `organizationId` (independent of whatever org is currently active in the session); 404 if not found or caller isn't a member of its org (same status for both, to avoid leaking existence).
  - `PATCH /api/projects/{id}` — partial update of name/slug/description; same authorization as GET; 409 on slug conflict.
  - `DELETE /api/projects/{id}` — hard delete; same authorization as GET.
- **Access control**: any organization member (owner/admin/member — Better Auth's `organization` plugin is currently used with zero custom role/permission config) may perform all Project CRUD operations. Explicit least-privilege trade-off, provisional for MVP.

## Testing Decisions

- A good test here asserts **observable HTTP behavior** of the route handlers (status codes, response bodies, persisted DB state) — not internal calls to helpers or Drizzle query-builder internals.
- **Single seam**: import and invoke the exported `GET`/`POST`/`PATCH`/`DELETE` functions from the route handler files directly with constructed `Request` objects, run against a real Postgres test database (reusing the existing `docker-compose.yml` service) and real Better-Auth-issued sessions (created via the actual `auth.api.signUpEmail` + `auth.api.createOrganization` flows — not mocked).
- No isolated unit tests for the new session helper or zod schemas — both are exercised only through the route-handler seam.
- Modules under test (transitively, through that one seam): the two projects route handler files, the project schema, the new session helper, and the new validation module.
- **Prior art: none.** This repo currently has zero automated tests (per AGENTS.md). This work introduces **Vitest** as the test runner — a new dependency and config, the first of its kind in the repo.
- Each test creates and tears down its own user/org/session rows rather than relying on shared fixtures, since no seeding convention exists yet either.

## Out of Scope

- Modeling Resource or Application as database entities, or any foreign key/proto wiring between them and Project (explicitly deferred).
- Changes to `proto/resources/v1/resources.proto`, or to the still-stubbed `ocel dev`/`ocel init` CLI commands, to reference a `projectId`.
- Role/permission-based restrictions on Project CRUD beyond "any org member" — Better Auth's `organization` plugin role/permission configuration remains untouched.
- Pagination on the list endpoint.
- Soft delete / archival of Projects.
- Any dashboard UI/pages for managing Projects — this PRD covers the API layer only.
- Any change to `organization.slug`'s existing global-uniqueness behavior.
- CLI changes to read/write a local `projectId` config value.

## Further Notes

- Three ADRs should be recorded alongside implementation: (1) UUIDv7 over default-v4 for Project IDs, (2) by-id Project routes authorize against the project's real organization rather than the session's active organization, (3) any-org-member write access to Projects as a provisional MVP permission model, flagged for revisit once org roles/permissions are configured.
- A repo-root `CONTEXT.md` (the first one in this repo) should gain glossary entries for **Organization**, **Project**, and **Active Organization** as part of this work.
- This PRD was produced via a `/grilling` + `/domain-modeling` session; every field, relation, and access-control decision above was explicitly confirmed with the user rather than assumed.
